ARQUITECTURA DEL STACK (igual patrón que VGD)
---------------------------------------------
Docker Compose (docker-compose.prod.yml) con 4 servicios:
  - db        → PostgreSQL 16 + pgvector  (base de datos PROPIA de Nexus)
  - backend   → FastAPI + uvicorn (chat API, streaming SSE, /health)
  - frontend  → Next.js standalone (UI; el origen público se hornea al build)
  - nginx     → reverse proxy interno; ÚNICO servicio publicado, en
                127.0.0.1:<NGINX_HOST_PORT> (solo localhost, no expuesto)

IIS actúa como reverse proxy externo (ARR + URL Rewrite, ya instalados):
  Usuario → IIS :<PUERTO_HTTPS_LIBRE> (TLS, cert de app-nexusqtech.com)
          → 127.0.0.1:<NGINX_HOST_PORT> (nginx de Nexus)
          → frontend:3000 / backend:8000
  backend → GCP / Gemini (internet)

Como el navegador solo habla con UN origen (https://app-nexusqtech.com:<puerto>)
y nginx enruta /api internamente, NO hay CORS entre dominios.

DIFERENCIAS CLAVE RESPECTO A VGD
--------------------------------
  - Nexus trae su PROPIA base de datos pgvector en Docker. NO se conecta al
    PostgreSQL nativo del servidor → NO hay que tocar pg_hba.conf.
  - Nexus NO usa Alembic. Al arrancar, el backend crea el esquema y aplica
    micro-migraciones automáticamente, y SIEMBRA el primer admin desde
    ADMIN_EMAIL / ADMIN_PASSWORD. No hay comandos manuales de migración/seed.
  - Embeddings en local (all-MiniLM-L6-v2, 384 dims); el modelo se descarga en
    el primer arranque (requiere internet). El LLM sigue siendo Gemini (Vertex).

PROBLEMAS Y SOLUCIONES
----------------------
- IIS 502 Bad Gateway:
    El stack no está corriendo o el puerto del web.config no coincide con
    NGINX_HOST_PORT. Verificar docker compose ps y la URL del web.config.

- La respuesta del chat NO llega en streaming (aparece toda de golpe):
    ARR está bufferizando. nginx ya tiene proxy_buffering off; si persiste,
    el buffer de respuesta de ARR es global y lo administra el dueño del
    servidor. Consultar con el administrador ANTES de cambiar ajustes
    globales de ARR (afectan a VGD y demás apps).

- Nadie puede iniciar sesión:
    Revisar que ADMIN_EMAIL/ADMIN_PASSWORD estaban en .env en el primer
    arranque. Si se cambiaron después, recrear el usuario o borrar el volumen
    pgdata SOLO si la base está vacía (¡cuidado en prod!).

- Error de cert (ERR_CERT_*) o HTTPS cuelga:
    El binding HTTPS del sitio Nexus no tiene cert válido para
    app-nexusqtech.com. Reasignar (Paso 7/8).

- backend no arranca:
    docker compose -f docker-compose.prod.yml logs backend
    Causas comunes: .env incompleto, JWT_SECRET débil, falta
    gcp-credentials.json, o sin internet para Gemini/descarga del modelo.

- Puerto elegido "EN USO":
    Elegir otro libre y actualizar .env (NGINX_HOST_PORT/PUBLIC_ORIGIN) y/o el
    binding IIS y el web.config. NUNCA apropiarse del puerto de otra app.

NOTAS
-----
- Datos persistentes en volúmenes Docker: pgdata (BD) y nexus_data (archivos
  subidos). Sobreviven reinicios; solo se pierden con "down -v".

- /docs y /redoc están bloqueados por nginx en este despliegue.
